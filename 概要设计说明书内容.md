# 概要设计说明书

## 1 引言

### 1.1 编写目的

本概要设计说明书旨在对文言文学习平台（wenyan-platform）系统进行总体设计说明，明确系统的整体架构、模块划分、接口设计、数据结构设计及出错处理设计，为后续的详细设计和编码实现提供依据。

预期读者包括：项目开发人员、测试人员、项目管理人员及相关技术评审人员。

### 1.2 背景

- **待开发软件系统的名称**：文言文学习平台（wenyan-platform）
- **项目任务提出者**：教学研究团队
- **开发者**：项目开发团队
- **用户**：中小学学生、语文教师
- **运行环境**：支持现代浏览器的PC端及移动端设备

本系统是一个面向文言文教学的在线学习平台，旨在通过数字化手段提升学生的文言文学习效率和兴趣，为教师提供教学辅助工具。

### 1.3 定义

| 术语 | 定义 |
|------|------|
| 文言文 | 中国古代的书面语言，与白话文相对 |
| Level1 | 一级学习阶段，侧重基础字词理解 |
| Level2 | 二级学习阶段，侧重对话与阅读理解 |
| Level3 | 三级学习阶段，侧重情景应用与自适应学习 |
| 多角色朗读 | 由不同角色配音的课文朗读形式 |
| 自适应测验 | 根据答题情况动态调整题目难度的测验形式 |
| SPA | 单页应用（Single Page Application） |
| API | 应用程序编程接口（Application Programming Interface） |

## 2 总体设计

### 2.1 需求规定

#### 2.1.1 功能需求

1. **课文展示功能**：展示文言文原文、字词注释、作者背景等基础信息
2. **多角色朗读功能**：提供分角色的课文朗读音频，支持播放控制
3. **分级学习功能**：
   - Level1：基础字词测验
   - Level2：对话理解与测验
   - Level3：情景文本与自适应测验
4. **文化卡片功能**：展示与课文相关的文化背景知识
5. **学生管理功能**：学生注册、信息查询与更新
6. **答题记录功能**：记录学生答题情况，支持按学生或课文查询
7. **登录认证功能**：学生登录验证

#### 2.1.2 性能需求

1. 页面加载时间：首屏加载时间不超过3秒
2. 数据查询响应时间：API接口响应时间不超过500ms
3. 并发支持：支持至少100名学生同时在线学习
4. 音频播放：支持流畅播放，无明显卡顿

#### 2.1.3 输入输出项目

**主要输入项目**：
- 学生学号、姓名
- 答题答案（单选/多选）
- 课文ID查询参数
- 分页查询参数

**主要输出项目**：
- 课文基础信息（标题、作者、朝代、原文）
- 字词注释列表
- 多角色朗读数据
- 各级测验题目及选项
- 学生信息及答题记录
- 得分统计结果

### 2.2 运行环境

#### 2.2.1 硬件环境

- **服务器端**：
  - CPU：双核及以上
  - 内存：4GB及以上
  - 硬盘：20GB及以上可用空间
  - 网络：100Mbps及以上带宽

- **客户端**：
  - 支持现代浏览器的PC或移动设备
  - 具备音频播放能力

#### 2.2.2 软件环境

- **服务器端**：
  - 操作系统：Linux/Windows
  - 运行环境：Node.js 20.19.0 及以上
  - 数据库：SQLite 3
  - Web服务器：Express.js

- **客户端**：
  - 浏览器：Chrome、Firefox、Edge、Safari等现代浏览器
  - 支持HTML5、CSS3、JavaScript ES6+

### 2.3 基本设计概念和处理流程

#### 2.3.1 设计概念

本系统采用前后端分离的架构设计：
- **前端**：采用Vue 3 + TypeScript + Vite技术栈，实现单页应用（SPA）
- **后端**：采用Node.js + Express.js技术栈，提供RESTful API服务
- **数据存储**：采用SQLite数据库存储学生信息和答题记录，JSON文件存储课文数据

#### 2.3.2 整体处理流程

```
用户访问前端页面
    ↓
前端路由匹配页面组件
    ↓
页面组件调用API服务请求数据
    ↓
后端接收请求，进行路由匹配
    ↓
控制器处理请求，调用服务层
    ↓
服务层访问数据库或JSON文件
    ↓
返回数据给前端
    ↓
前端渲染页面展示给用户
    ↓
用户交互（答题、播放音频等）
    ↓
前端提交数据到后端
    ↓
后端处理并存储数据
    ↓
返回处理结果
```

#### 2.3.3 学习流程

```
首页（选择课文）
    ↓
规则页（学习引导）
    ↓
StepOne（一级学习：字词+基础测验）
    ↓
StepTwo（二级学习：对话+二级测验）
    ↓
StepThree（三级学习：情景+自适应测验）
    ↓
详情页（学习结果展示）
```

### 2.4 结构

2. **业务逻辑层**：后端Express服务，负责业务逻辑处理
3. **数据访问层**：SQLite数据库 + JSON文件，负责数据存储

### 2.6 人工处理过程

本系统为自动化学习系统，主要学习过程由系统自动完成。以下情况需要人工处理：

1. **课文数据录入**：需要人工整理和录入课文原文、字词注释、测验题目等数据
2. **音频录制**：多角色朗读音频需要人工录制
3. **学生账号管理**：批量学生账号创建可由管理员人工操作
4. **系统维护**：服务器维护、数据备份等需要人工操作
5. **问题答疑**：学生学习过程中遇到的问题需要教师人工解答

### 2.7 尚未解决的问题

1. **大规模并发支持**：当前设计基于SQLite，对于大规模并发场景可能存在性能瓶颈，后续可考虑迁移到MySQL等关系型数据库
2. **富媒体内容支持**：当前主要支持文本、音频、视频，对于交互式动画等富媒体内容支持有限
3. **学习数据分析**：当前仅记录答题数据，深度的学习行为分析和学习路径推荐尚未实现
4. **移动端适配优化**：虽然支持移动端访问，但针对移动端的交互体验优化仍需加强
5. **教师端功能**：当前主要面向学生端，教师端的教学管理、作业布置、成绩分析等功能有待完善

## 3 接口设计

### 3.1 用户接口

#### 3.1.1 界面形式

系统采用Web图形用户界面（GUI），基于Vue 3构建的单页应用，支持响应式布局，可适配PC端和移动端。

#### 3.1.2 主要页面及功能

| 页面名称 | 路由路径 | 主要功能 |
|---------|---------|---------|
| 首页 | / | 展示课文列表，供用户选择学习内容 |
| 规则页 | /rules/:id | 展示学习规则和引导视频 |
| 一级学习页 | /stepone/:id | 一级学习内容：字词注释、多角色朗读、基础测验 |
| 二级学习页 | /steptwo/:id | 二级学习内容：对话理解、二级测验 |
| 三级学习页 | /stepthree/:id | 三级学习内容：情景文本、自适应测验 |
| 详情页 | /detail/:id | 展示学习结果和详细信息 |
| 答题查询页 | /answer-query | 查询学生答题记录和统计 |
| 404页面 | * | 页面不存在时的提示页面 |

#### 3.1.3 用户操作方式

- **鼠标操作**：点击、悬停、滚动
- **键盘操作**：Tab键切换焦点、Enter键确认
- **触摸操作**：移动端支持点击、滑动等触摸操作
- **音频控制**：播放、暂停、进度调整、音量控制

### 3.2 外部接口

#### 3.2.1 硬件接口

系统不直接与硬件交互，通过浏览器提供的标准API间接访问硬件资源：
- 音频播放：通过HTML5 Audio API
- 网络通信：通过浏览器的HTTP请求
- 本地存储：通过浏览器的LocalStorage、SessionStorage、IndexedDB

#### 3.2.2 软件接口

**前端与后端接口**：
- 协议：HTTP/HTTPS
- 数据格式：JSON
- 认证方式：可选的Token认证（JWT）

**后端与数据库接口**：
- 数据库：SQLite 3
- 驱动：sqlite3 Node.js模块
- 通信方式：本地文件访问

**后端与文件系统接口**：
- 数据文件：JSON格式文件
- 媒体文件：MP3音频、MP4视频
- 访问方式：本地文件系统读取

### 3.3 内部接口

- Services层：统一的API调用接口

**模块间接口**：
- 配置模块：提供统一的配置读取接口

- 工具模块：提供通用工具函数接口

## 4 运行设计

### 4.1 运行模块组合

#### 4.1.1 学生学习场景

**触发条件**：学生访问学习页面
**运行模块组合**：
```
前端：
  HomeView → RuleView → StepOneView → StepTwoView → StepThreeView → DetailView
  调用组件：WordList、MultiRoleReading、Level1Quiz、DialogText、Level2Quiz等
  调用服务：apiService、useDataLoader、useAnswerSubmitter等

后端：
  textsController → textsService → JSON数据文件
  answerController → answerService → answer_records.db
  studentController → studentService → students.db
```

#### 4.1.2 答题提交场景

**触发条件**：学生提交答案
**运行模块组合**：
```
前端：
  useAnswerSubmitter → apiService.submitAnswers
  调用组件：Question、Options、QuizCard等

后端：
  answerController.submitAnswers → answerService.submitAnswers
  → 数据库操作：answer_records.db
  → 可选：studentService.createOrUpdateStudent（自动注册学生）
```

#### 4.1.3 答题查询场景

**触发条件**：查询答题记录
**运行模块组合**：
```
前端：
  AnswerQueryView → apiService.getAnswersByStudentId / getAnswersByWenId

后端：
  answerController → answerService → answer_records.db + students.db
```

#### 4.1.4 学生登录场景

**触发条件**：学生登录
**运行模块组合**：
```
前端：
  LoginModal → apiService.login → authStore

后端：
  authController.login → studentService → students.db
  → token生成 → 返回前端
```

### 4.2 运行控制

#### 4.2.1 前端运行控制

**路由控制**：
- 基于Vue Router的前端路由控制
- 路由守卫：认证守卫（requiresAuth）、全局前置守卫
- 动态路由：支持路由参数传递

**状态控制**：
- Pinia状态管理：全局状态集中管理
- 组件状态：组件内部响应式状态
- 本地存储：持久化状态存储

**流程控制**：
- 学习流程：按预设步骤顺序进行
- 答题流程：逐题或批量提交
- 错误处理：统一的错误捕获和展示

#### 4.2.2 后端运行控制

**请求控制**：
- Express中间件链：请求预处理
- 路由分发：根据URL和HTTP方法分发到对应控制器
- 限流控制：防止恶意请求和刷接口

**业务控制**：
- 事务控制：数据库操作的原子性保证
- 并发控制：serialize保证数据库操作顺序
- 数据校验：输入参数合法性校验

**错误控制**：
- 全局错误处理中间件：统一错误响应
- 404处理：未匹配路由处理
- 异常捕获：try-catch捕获业务异常

### 6.2 补救措施

#### 6.2.1 后备技术

**数据备份**：
- 数据库文件定期备份（每日备份）
- JSON数据文件版本管理（Git）
- 媒体文件多副本存储

**服务降级**：
- 后端服务不可用时，前端可降级使用本地静态数据
- 部分功能故障时，核心学习功能仍可正常使用

**缓存机制**：
- 前端数据缓存（dataCache）
- 浏览器本地存储（LocalStorage）
- 后端内存缓存（频繁访问的数据）

#### 6.2.2 降效技术

**功能降级**：
- 音频加载失败时，提供文本阅读模式
- 视频加载失败时，提供图文说明
- 自适应测验不可用时，提供固定难度测验

**性能降级**：
- 高并发时，减少非必要查询
- 降低数据实时性，增加缓存时间
- 限制同时在线人数（极端情况）

#### 6.2.3 恢复及再启动技术

**前端恢复**：
- 页面刷新自动重新加载
- 错误重试机制（自动重试1-3次）
- 手动重试按钮

**后端恢复**：
- 进程守护：PM2进程管理，自动重启
- 优雅关闭：SIGINT信号处理，正常关闭连接
- 数据库恢复：SQLite事务保证数据一致性
