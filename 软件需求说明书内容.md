# 需求说明书

## 1 引言

### 1.1 编写的目的

本需求说明书旨在对文言文学习平台（wenyan-platform）系统的功能需求、性能需求、数据需求、运行环境需求等进行详细说明，明确系统的功能边界和技术指标，为系统的设计、开发、测试和验收提供依据。

预期读者包括：项目管理人员、产品经理、开发人员、测试人员、教师用户代表。

### 1.2 背景

- **待开发的系统的名称**：文言文学习平台（wenyan-platform）
- **本项目的任务提出者**：教学研究团队
- **开发者**：项目开发团队
- **用户**：中小学学生、语文教师、教学管理人员
- **运行环境**：支持现代浏览器的PC端及移动端设备

本系统是一个面向文言文教学的在线学习平台，旨在通过数字化手段提升学生的文言文学习效率和学习兴趣，为教师提供教学辅助工具，为教学管理提供数据支持。

### 1.3 定义

| 术语 | 定义 |
|------|------|
| 文言文 | 中国古代的书面语言，与白话文相对，是语文教学的重要内容 |

## 2 任务概述

### 2.1 目标

本系统的开发目标是构建一个面向中小学文言文教学的在线学习平台，具体目标如下：

1. **提升学习效率**：通过数字化手段，将传统文言文学习内容转化为交互式学习资源，提升学生的学习效率
2. **增强学习兴趣**：通过多角色朗读、分级测验、文化卡片等丰富的学习形式，增强学生的学习兴趣
3. **个性化学习**：通过自适应测验等功能，实现个性化学习路径，满足不同水平学生的需求
4. **教学辅助**：为教师提供教学资源和学生学习数据，辅助教学决策
5. **数据记录**：完整记录学生的学习过程和答题数据，为教学研究提供数据支持

**作用范围**：
- 主要面向中小学文言文教学场景
- 支持课内学习和课外自主学习
- 支持学生端学习和教师端管理（教师端为后续扩展功能）

**与其他系统的关系**：
- 本系统为独立的Web应用，不依赖其他业务系统
- 可与学校的教务管理系统进行数据对接（后续扩展）
- 支持通过浏览器直接访问，无需安装客户端

### 2.2 用户的特点

本系统的最终用户主要包括以下几类：

**1. 学生用户**
- 教育水平：中小学生，年龄约10-15岁
- 技术专长：具备基本的电脑或手机操作能力，熟悉浏览器使用
- 使用频度：日常学习使用，预计每周使用3-5次，每次使用20-40分钟
- 特点：对新鲜事物接受度高，喜欢互动性强的学习方式；注意力集中时间有限，需要趣味性设计

**2. 教师用户**
- 教育水平：大学本科及以上，语文教师
- 技术专长：具备基本的信息技术应用能力
- 使用频度：备课和教学分析时使用，预计每周使用2-3次
- 特点：关注教学资源的质量和学生的学习数据

**3. 系统管理员**
- 教育水平：大专及以上，信息技术相关专业
- 技术专长：具备系统管理和维护能力
- 使用频度：日常维护，不定期使用
- 特点：关注系统的稳定性、安全性和可维护性

### 2.3 假定和约束

**开发假定**：
1. 假定用户具备基本的计算机操作能力和网络使用能力
2. 假定用户设备能够运行现代浏览器
3. 假定学校能够提供稳定的网络环境
4. 假定课文数据和音频资源能够按时准备完毕

**开发约束**：
1. **技术约束**：采用前后端分离架构，前端使用Vue 3，后端使用Node.js + Express，数据库使用SQLite
2. **进度约束**：按照项目计划的时间节点完成开发
3. **资源约束**：开发团队规模有限，需合理安排开发任务
4. **兼容性约束**：需支持主流浏览器（Chrome、Firefox、Edge、Safari）
5. **安全约束**：学生信息需妥善保护，符合个人信息保护相关法规
6. **性能约束**：需支持至少100名学生同时在线学习

## 3 需求规定

### 3.1 对功能的规定

#### 3.1.1 课文展示功能

**功能描述**：展示文言文原文、字词注释、作者背景、朝代背景等基础信息。

**输入**：
- 课文ID
- 用户选择的课文

**处理**：
- 根据课文ID从数据文件中读取课文基础信息
- 解析原文内容和字词注释
- 渲染展示到页面

**输出**：
- 课文标题、作者、朝代
- 文言文原文
- 字词注释列表（点击可展开详细释义）
- 作者背景介绍
- 朝代背景介绍

**容量**：
- 支持至少50篇课文
- 每篇课文字词注释不少于20个

#### 3.1.2 多角色朗读功能

**功能描述**：提供分角色的课文朗读音频，支持播放控制，增强学习趣味性。

**输入**：
- 课文ID
- 用户播放/暂停操作
- 用户进度调整操作
- 用户音量调整操作

**处理**：
- 加载音频文件
- 解析段落时间戳数据
- 播放音频并同步高亮当前段落
- 响应用户的播放控制操作

**输出**：
- 音频播放
- 播放进度条
- 当前朗读段落高亮
- 角色名称显示
- 播放时间显示

**容量**：
- 每篇课文支持至少一个多角色朗读音频
- 每篇课文段落不少于10段

#### 3.1.3 分级学习功能

**功能描述**：将学习内容分为三个级别，循序渐进地引导学生学习。

**Level1（基础级）**：
- 功能：字词学习 + 基础测验
- 输入：学生答题答案
- 处理：展示字词注释、播放朗读、出题测验、自动判分
- 输出：字词注释、朗读音频、测验题目、得分结果

**Level2（进阶级）**：
- 功能：对话理解 + 二级测验
- 输入：学生答题答案
- 处理：展示对话内容、出题测验、自动判分
- 输出：对话内容、测验题目、得分结果

**Level3（提高级）**：
- 功能：情景文本 + 自适应测验
- 输入：学生答题答案
- 处理：展示情景文本、根据答题情况动态调整题目难度、自动判分
- 输出：情景文本、自适应测验题目、得分结果

**容量**：
- 每篇课文每级测验不少于10道题
- 自适应测验支持至少3个难度等级

#### 3.1.4 文化卡片功能

**功能描述**：展示与课文相关的文化背景知识，拓展学生的文化视野。

**输入**：
- 课文ID
- 用户翻页/切换操作

**处理**：
- 读取文化卡片数据
- 渲染卡片内容
- 支持卡片切换

**输出**：
- 文化卡片标题
- 文化知识内容
- 相关图片（可选）

**容量**：
- 每篇课文不少于3张文化卡片

#### 3.1.5 学生管理功能

**功能描述**：管理学生账号信息，支持学生注册、信息查询和更新。

**输入**：
- 学生学号
- 学生姓名
- 班级信息

**处理**：
- 验证学号格式（纯数字）
- 检查学号是否已存在
- 保存/更新学生信息
- 查询学生信息

**输出**：
- 注册成功/失败提示
- 学生信息
- 学生列表

**容量**：
- 支持至少1000名学生注册
- 支持按班级筛选学生

#### 3.1.6 答题记录功能

**功能描述**：记录学生的答题情况，支持按学生或按课文查询，统计学习数据。

**输入**：
- 学生ID
- 课文ID
- 题目ID
- 用户答案
- 正确答案

**处理**：
- 比较用户答案和正确答案
- 计算得分（正确100分，错误0分）
- 保存答题记录
- 统计正确率和平均分
- 按条件查询和分组统计

**输出**：
- 答题提交结果（正确/错误、得分）
- 答题记录列表
- 统计数据（总题数、正确数、错误数、总分、平均分）

**容量**：
- 支持至少10万条答题记录
- 支持按学生、按课文、按时间等多维度查询

#### 3.1.7 登录认证功能

**功能描述**：学生登录验证，保护学生数据安全。

**输入**：
- 学生学号
- 学生姓名

**处理**：
- 验证学号和姓名是否匹配
- 生成认证令牌（JWT）
- 返回认证结果

**输出**：
- 登录成功/失败提示
- 认证令牌
- 学生信息

#### 3.1.8 学习引导功能

**功能描述**：通过规则页和引导视频，引导学生了解学习流程和方法。

**输入**：
- 课文ID
- 用户的下一步操作

**处理**：
- 展示学习规则
- 播放引导视频
- 引导学生进入学习流程

**输出**：
- 规则说明文字
- 引导视频
- 下一步操作按钮

### 3.2 对性能的规定

#### 3.2.1 精度

**数据精度**：
- 学生学号：纯数字字符串，长度不限
- 得分：整数，范围0-100分
- 时间：精确到秒
- 统计数据：整数或保留1位小数

**传输精度**：
- 数据传输过程中保证完整性，不丢失、不错位
- 答案提交保证准确性，不错题、不丢题

#### 3.2.2 时间特性要求

**页面加载**：
- 首页加载时间：≤3秒（正常网络环境）
- 学习页面加载时间：≤2秒
- 页面切换时间：≤500毫秒

**数据查询**：
- 课文列表查询响应时间：≤500毫秒
- 单篇课文数据查询响应时间：≤300毫秒
- 答题记录查询响应时间：≤500毫秒

**答题提交**：
- 单题提交响应时间：≤500毫秒
- 批量提交（20题）响应时间：≤1秒

**音频视频**：
- 音频启动时间：≤500毫秒
- 视频启动时间：≤2秒
- 播放过程流畅，无明显卡顿

**并发性能**：
- 支持至少100名学生同时在线学习
- 支持至少50名学生同时提交答案

#### 3.2.3 灵活性

**功能灵活性**：
- 课文数据通过JSON文件配置，新增课文无需修改代码
- 测验题目可通过数据文件灵活配置
- 学习流程支持配置化调整

**接口灵活性**：
- API接口采用RESTful风格，便于扩展
- 数据格式采用JSON，便于解析和转换
- 支持前后端独立部署和升级

**部署灵活性**：
- 支持Linux和Windows服务器部署
- 支持单机部署和分布式部署（后续扩展）
- 数据库支持SQLite，可迁移至MySQL等关系型数据库

**适配灵活性**：
- 支持PC端和移动端访问
- 支持主流浏览器（Chrome、Firefox、Edge、Safari）
- 响应式布局，适配不同屏幕尺寸

### 3.3 输入输出要求

#### 3.3.1 输入数据

**学生信息输入**：
- 数据类型：文本
- 媒体：键盘输入
- 格式：学号（纯数字）、姓名（中文）、班级（数字）
- 数值范围：学号长度不限，姓名2-20字符
- 精度：无

**答题答案输入**：
- 数据类型：单选/多选
- 媒体：鼠标点击/触摸点击
- 格式：选项索引或选项值
- 数值范围：根据题目选项数量而定
- 精度：无

**查询参数输入**：
- 数据类型：文本/数字
- 媒体：键盘输入/选择
- 格式：课文ID、学生ID、页码等
- 数值范围：根据实际数据
- 精度：无

#### 3.3.2 输出数据

**课文信息输出**：
- 数据类型：文本
- 媒体：屏幕显示
- 格式：结构化文本
- 数值范围：无
- 精度：无

**测验题目输出**：
- 数据类型：文本 + 选项
- 媒体：屏幕显示
- 格式：题目 + 选项列表
- 数值范围：无
- 精度：无

**答题结果输出**：
- 数据类型：文本 + 数字
- 媒体：屏幕显示
- 格式：正确/错误标识、得分、统计数据
- 数值范围：得分0-100，正确率0%-100%
- 精度：得分整数，正确率保留1位小数

**统计数据输出**：
- 数据类型：数字
- 媒体：屏幕显示
- 格式：总数、正确数、错误数、平均分等
- 数值范围：根据实际数据
- 精度：整数或保留1位小数

#### 3.3.3 控制输出

**播放控制输出**：
- 播放/暂停状态
- 播放进度
- 音量大小
- 当前段落高亮

**导航控制输出**：
- 页面跳转
- 学习步骤切换
- 返回上一级

**状态提示输出**：
- 加载状态
- 成功/失败提示
- 错误信息
- 确认对话框

### 3.4 数据管理能力要求

#### 3.4.1 数据存储规模

**学生数据**：
- 学生表记录数：预计1000-5000条
- 每条记录大小：约100字节
- 总存储量：约0.5-2.5MB

**答题记录数据**：
- 答题记录表记录数：预计10万-100万条
- 每条记录大小：约500字节
- 总存储量：约50-500MB

**课文数据**：
- 课文篇数：预计50-200篇
- 每篇基础数据：约10KB
- 每篇字词注释：约20KB
- 每篇测验数据：约30KB
- 总存储量：约3-12MB（不含音视频）

**音视频数据**：
- 每篇课文音频：约2-5MB
- 引导视频：约10-20MB
- 总存储量：根据课文数量而定

**总存储估算**：
- 数据文件：约100MB
- 音视频文件：约1-5GB
- 数据库文件：约100-500MB
- 总计：约1.5-6GB

#### 3.4.2 数据增长估算

**学生数据增长**：
- 每年新增学生：约1000人
- 每年增长存储：约0.1MB

**答题记录增长**：
- 每生每学期答题：约500道
- 每年答题记录：约50万-100万条
- 每年增长存储：约250-500MB

**课文数据增长**：
- 每年新增课文：约20-50篇
- 每年增长存储：约100-500MB（含音视频）

**可预见的增长**：
- 系统运行3年，数据总量预计达到5-15GB
- 需考虑数据备份和归档策略

#### 3.4.3 数据管理要求

**数据备份**：
- 数据库每日备份
- 数据文件定期备份
- 备份数据保留至少30天

**数据安全**：
- 学生信息加密存储（可选）
- 访问日志记录
- 防止SQL注入等安全漏洞

**数据完整性**：
- 数据库事务保证数据一致性
- 数据校验机制
- 异常数据处理

### 3.5 故障处理要求

#### 3.5.1 软件故障

**前端页面故障**：
- 故障现象：页面白屏、功能异常、脚本错误
- 故障后果：用户无法正常使用功能
- 处理要求：
  - 页面刷新后可恢复
  - 关键操作有错误提示
  - 错误日志记录便于排查

**后端服务故障**：
- 故障现象：服务无响应、接口报错
- 故障后果：前端无法获取数据、无法提交答案
- 处理要求：
  - 服务自动重启（PM2进程守护）
  - 前端有降级方案（使用本地数据）
  - 错误日志详细记录

**数据库故障**：
- 故障现象：数据库连接失败、数据读写错误
- 故障后果：无法读写学生信息和答题记录
- 处理要求：
  - 数据库文件损坏可从备份恢复
  - 写入操作保证原子性
  - 错误及时告警

#### 3.5.2 硬件故障

**服务器故障**：
- 故障现象：服务器宕机、网络中断
- 故障后果：系统完全不可用
- 处理要求：
  - 有备用服务器或快速恢复方案
  - 数据定期备份，可快速恢复

**网络故障**：
- 故障现象：网络延迟高、断网
- 故障后果：页面加载慢、数据提交失败
- 处理要求：
  - 前端有加载状态提示
  - 提交失败有重试机制
  - 网络恢复后可继续使用

**客户端设备故障**：
- 故障现象：浏览器崩溃、设备死机
- 故障后果：用户学习中断
- 处理要求：
  - 答题进度不丢失（后端已保存）
  - 重新登录后可继续学习

#### 3.5.3 故障恢复要求

**恢复时间目标（RTO）**：
- 一般故障：≤10分钟
- 严重故障：≤2小时
- 灾难性故障：≤24小时

**恢复点目标（RPO）**：
- 数据丢失不超过24小时（每日备份）
- 答题数据实时保存，不丢失已提交的答案

- 性能监控（可选）

#### 4.4.3 控制流程

**启动流程**：
1. 加载环境变量和配置
2. 连接数据库（初始化表结构）
3. 注册中间件和路由
4. 启动HTTP服务
5. 监听指定端口
6. 记录启动日志

**运行流程**：
1. 接收HTTP请求
2. 执行中间件链（日志、CORS、解析等）
3. 匹配路由
4. 执行业务逻辑
5. 访问数据库或文件
6. 返回响应
7. 记录访问日志

**停止流程**：
1. 接收停止信号
2. 停止接收新请求
3. 等待现有请求处理完成
4. 关闭数据库连接
5. 关闭HTTP服务
6. 记录停止日志
7. 进程退出
