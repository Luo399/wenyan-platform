## 1. 概述

### 1.1 目的

为确保 wenyan-platform 项目进入持续维护阶段后的变更操作规范化、安全化，保障系统稳定性和数据结构一致性，特制定本规范。

### 1.2 适用阶段

本规范适用于项目开发完成后的持续维护阶段，包括但不限于：
- Bug修复
- 数据内容更新
- 性能优化
- 安全补丁

### 1.3 核心原则

| 原则 | 说明 |
|-----|------|
| **变更安全原则** | 在确保不引入新缺陷的前提下实施变更，执行三级风险评估 |
| **数据结构一致性** | 所有改动需与当前生产版本进行自动化+人工双重数据结构对比分析 |
| **内容优先原则** | 严格限定于对具体数据内容进行修改，严禁变更数据框架或结构 |

## 4. 核心流程与规范

### 4.1 变更申请与审批流程

#### 4.1.1 流程图

```
变更申请 → 风险评估 → 技术评审 → 审批 → 实施 → 测试 → 上线 → 验收
    ↑                          |
    └──────────────────────────┘
                         ↓
                    拒绝/退回修改
```

#### 4.1.2 流程说明

| 阶段 | 职责角色 | 说明 | 输出 |
|-----|---------|------|-----|
| **变更申请** | 开发工程师 | 提交变更请求单，描述变更内容、原因、影响范围 | 变更请求单 |
| **风险评估** | 技术负责人 | 执行三级风险评估（功能、兼容性、性能） | 风险评估报告 |
| **技术评审** | 技术评审委员会 | 评审变更方案技术可行性 | 评审意见 |
| **审批** | 项目经理 | 审批变更请求 | 审批意见 |
| **实施** | 开发工程师 | 执行变更修改 | 修改代码/数据 |
| **测试** | 测试工程师 | 执行单元测试、回归测试 | 测试报告 |
| **上线** | 运维工程师 | 部署到生产环境 | 部署记录 |
| **验收** | 产品负责人 | 确认变更符合预期 | 验收确认单 |

#### 4.1.3 各角色职责

| 角色 | 职责 |
|-----|------|
| **开发工程师** | 提交变更申请、实施修改、编写单元测试 |
| **技术负责人** | 风险评估、技术方案审核 |
| **技术评审委员会** | 复杂变更的技术评审 |
| **项目经理** | 变更优先级评估、审批决策 |
| **测试工程师** | 测试执行、回归测试、报告编写 |
| **运维工程师** | 部署上线、环境管理 |
| **产品负责人** | 业务验收、需求确认 |

### 4.2 数据结构检查标准操作程序(SOP)

#### 4.2.1 检查内容

| 检查项 | 检查内容 | 检查方法 |
|-------|---------|---------|
| **属性名称匹配** | 属性名称精确匹配 | 自动化对比 |
| **属性数量完整性** | 字段数量与预期一致 | 自动化统计 |
| **数据类型一致性** | 各字段类型符合定义 | 自动化验证 |
| **层级结构验证** | 嵌套关系正确 | 自动化分析 |
| **约束条件兼容性** | 默认值、枚举值等约束 | 人工+自动化 |

#### 4.2.2 操作流程

```
1. 提取生产环境数据结构定义
2. 获取待变更数据的结构定义
3. 运行自动化验证工具（data_validator.py）
4. 人工复核验证结果
5. 生成数据结构对比报告
6. 确认无误后进入变更实施阶段
```

#### 4.2.3 验证工具使用

```bash
# 运行数据结构验证器
cd 数据/maintenance_tools
python data_validator.py

# 验证报告输出路径
数据/maintenance_tools/validation_report.md
```

#### 4.2.4 检查报告模板


### 4.3 维护测试要求

#### 4.3.1 测试覆盖率要求

| 测试类型 | 覆盖率要求 |
|---------|----------|
| **单元测试** | ≥90% |
| **集成测试** | ≥80% |
| **回归测试** | 100%覆盖核心功能 |

#### 4.3.2 测试用例要求

| 测试阶段 | 要求 |
|---------|------|
| **单元测试** | 覆盖所有函数/方法，包含边界条件 |
| **集成测试** | 验证模块间接口正确性 |
| **回归测试** | 确保修改不影响现有功能 |
| **性能测试** | 关键接口响应时间满足要求 |

#### 4.3.3 测试执行流程

```
编写测试用例 → 执行单元测试 → 执行集成测试 → 执行回归测试 → 生成测试报告 → 评审通过
```

#### 4.3.4 测试报告要求

| 报告内容 | 要求 |
|---------|------|
| **测试概述** | 测试范围、目的、环境 |
| **测试用例** | 用例清单、执行结果 |
| **缺陷统计** | 缺陷数量、严重程度分布 |
| **性能指标** | 响应时间、吞吐量等 |
| **结论** | 测试是否通过、是否可上线 |

### 4.4 版本控制策略

#### 4.4.1 分支管理

| 分支类型 | 命名格式 | 用途 | 生命周期 |
|---------|---------|-----|---------|
| **主分支** | `main` | 稳定版本 | 永久 |
| **开发分支** | `develop` | 集成开发 | 永久 |
| **功能分支** | `feature/xxx` | 新功能开发 | 临时 |
| **修复分支** | `bugfix/xxx` | Bug修复 | 临时 |
| **发布分支** | `release/xxx` | 发布准备 | 临时 |
| **热修复分支** | `hotfix/xxx` | 线上紧急修复 | 临时 |

#### 4.4.2 分支管理流程

```
main ←─── merge ←─── release/vx.y.z ←─── merge ←─── develop
                ↑                              ↑
                │                              │
                └─── hotfix/xxx ───────────────┘
                              ↑
                              │
                     bugfix/xxx
                              ↑
                              │
                     feature/xxx
```

#### 4.4.3 提交规范

提交信息格式：
```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

| Type | 说明 |
|-----|------|
| **feat** | 新功能 |
| **fix** | Bug修复 |
| **docs** | 文档更新 |
| **style** | 代码风格 |
| **refactor** | 重构 |
| **test** | 测试 |
| **chore** | 构建/工具 |
| **perf** | 性能优化 |

#### 4.4.4 版本号命名规则

```
v<major>.<minor>.<patch>
```

| 部分 | 说明 | 触发条件 |
|-----|------|---------|
| **major** | 重大变更 | 不兼容升级 |
| **minor** | 新功能 | 向后兼容的功能新增 |
| **patch** | Bug修复 | 向后兼容的问题修复 |

### 4.6 紧急变更处理流程

#### 4.6.1 紧急变更定义

符合以下条件之一的变更为紧急变更：
- 生产环境系统故障导致服务不可用
- 安全漏洞需要立即修复
- 影响核心业务流程的严重Bug



#### 4.6.4 事后复盘

紧急变更完成后24小时内必须进行复盘：
- 问题根因分析
- 解决方案评估
- 预防措施制定
- 文档更新

## 5. 数据变更规范

### 5.1 允许的变更类型

| 类型 | 说明 | 示例 |
|-----|------|-----|
| **内容修改** | 修改数据内容值 | 修改题目答案、更新文本内容 |
| **数据补充** | 补充缺失的数据内容 | 完善字词注释 |
| **格式修正** | 修正数据格式问题 | 修复JSON格式错误 |

### 5.2 禁止私自修改的变更类型（发起会话请求的修改豁免）

| 类型 | 说明 | 示例 |
|-----|------|-----|
| **字段新增** | 新增数据字段 | 在JSON中添加新属性 |
| **字段删除** | 删除已有字段 | 移除JSON属性 |
| **类型变更** | 修改字段类型 | 将string改为number |
| **结构调整** | 调整数据层级结构 | 修改嵌套关系 |
